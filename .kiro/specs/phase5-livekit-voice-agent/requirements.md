# Requirements Document

## Introduction

Phase 5 implements the LiveKit Voice Agent and Post-Chat Summary system for JyotishAI. This phase repairs the broken agent pipeline in `agent/main.py` by replacing invalid plugin references with the correct `livekit-agents` v1.x framework imports, wires the agent's system prompt to the `detailed_report` narrative blocks already stored in the database, persists every conversation turn to `public.session_messages`, and triggers an automated post-session assessment generator when a LiveKit room finishes. The assessment is produced by OpenRouter and stored as structured JSONB in `public.sessions.assessment`.

## Glossary

- **Agent**: The `agent/main.py` Python process that connects to a LiveKit room and runs the STT → LLM → TTS voice pipeline.
- **AgentSession**: The `livekit.agents.AgentSession` class that orchestrates the voice pipeline in livekit-agents v1.x.
- **Celeste**: The AI astrologer persona presented to the user during a voice session.
- **chart_client**: The `agent/chart_client.py` module responsible for fetching chart data from the backend given a room name.
- **detailed_report**: The JSONB column in `public.birth_charts` containing the human-narrative text blocks (`personal`, `career`, `love` × `past`, `current`, `future`).
- **fetch_chart_data**: An async function in `chart_client.py` that accepts a room name and returns the full chart dict including `detailed_report`.
- **LiveKit Room**: A named virtual room managed by the LiveKit server; its name is stored in `public.sessions.livekit_room`.
- **OpenRouter**: The LLM gateway at `https://openrouter.ai/api/v1` used for all AI text generation.
- **Post-Session Assessment**: A structured JSON object (`session_summary`, `key_insights`, `action_items`) generated by OpenRouter after a session ends and stored in `public.sessions.assessment`.
- **room_finished**: The LiveKit webhook event type sent to `POST /token/webhook` when all participants leave a room.
- **session_messages**: The `public.session_messages` Supabase table that stores per-turn transcripts with `session_id`, `role`, and `content`.
- **VoicePipelineAgent**: The `livekit.agents.VoicePipelineAgent` class used in older livekit-agents API; replaced by `AgentSession` + `Agent` in v1.x.
- **Webhook Handler**: The `POST /token/webhook` route in `backend/app/routers/token.py` that receives LiveKit server events.

## Requirements

### Requirement 1

**User Story:** As a developer, I want the agent process to start cleanly without import errors, so that the voice pipeline is reliably available for every session.

#### Acceptance Criteria

1. WHEN the agent process starts, THE Agent SHALL import only symbols that exist in the installed `livekit-agents==1.5.x` package (`AgentSession`, `Agent`, `JobContext`, `WorkerOptions`) and the installed plugin packages (`livekit-plugins-deepgram`, `livekit-plugins-elevenlabs`, `livekit-plugins-silero`, `livekit-plugins-openai`).
2. WHEN the agent process starts, THE Agent SHALL NOT reference removed or non-existent symbols such as `inference.STT`, `inference.LLM`, `sarvam.TTS`, `MultilingualModel`, or `VoicePipelineAgent`.
3. IF the agent process encounters a missing environment variable for any required API key, THEN THE Agent SHALL log an error message identifying the missing variable and exit with a non-zero status code.

---

### Requirement 2

**User Story:** As a user, I want Celeste to speak directly about the narrative content I see on my dashboard, so that the voice session feels coherent and personalised.

#### Acceptance Criteria

1. WHEN a new room is joined, THE Agent SHALL fetch the chart record associated with that room name by calling `fetch_chart_data(room_name)` before constructing the system prompt.
2. WHEN building the system prompt, THE Agent SHALL embed the full `detailed_report` JSON object from the fetched chart record as the sole source of narrative context for Celeste.
3. WHILE a session is active, THE Agent SHALL instruct Celeste to respond using only the narrative text from `detailed_report` and to avoid reading out raw degree values, house indices, or sign names unless the user explicitly requests the technical breakdown.
4. WHEN the chart fetch fails, THE Agent SHALL fall back to a generic greeting that acknowledges the missing data and invites the user to re-enter their birth details.
5. WHEN the session starts, THE Agent SHALL deliver an opening greeting that addresses the user by their `full_name` from the chart record.

---

### Requirement 3

**User Story:** As a product owner, I want every spoken turn to be persisted to the database, so that the transcript is available for post-session analysis.

#### Acceptance Criteria

1. WHEN a user speech turn is committed by the STT pipeline, THE Agent SHALL insert a row into `public.session_messages` with `role = 'user'` and `content` equal to the committed transcript text.
2. WHEN an agent speech turn is committed by the TTS pipeline, THE Agent SHALL insert a row into `public.session_messages` with `role = 'assistant'` and `content` equal to the committed transcript text.
3. WHEN the session record cannot be resolved from the room name, THE Agent SHALL log a warning and skip message persistence without crashing the voice pipeline.
4. IF a database insert for a session message fails, THEN THE Agent SHALL log the error and continue the voice session without interruption.

---

### Requirement 4

**User Story:** As a user, I want to receive a structured summary of my session after it ends, so that I can review the key insights and action items from my consultation.

#### Acceptance Criteria

1. WHEN the Webhook Handler receives a `room_finished` event, THE Webhook Handler SHALL retrieve the matching session record from `public.sessions` using the room name.
2. WHEN the session record is found, THE Webhook Handler SHALL fetch all rows from `public.session_messages` for that `session_id`, ordered by `created_at` ascending.
3. WHEN the transcript contains at least one message, THE Webhook Handler SHALL send the full transcript to OpenRouter using model `openai/gpt-4o-mini` with `response_format: {"type": "json_object"}` and a timeout of 30 seconds.
4. WHEN OpenRouter returns a successful response, THE Webhook Handler SHALL parse the response into a `SessionAssessment` object containing `session_summary` (string), `key_insights` (list of strings), and `action_items` (list of strings).
5. WHEN the `SessionAssessment` is parsed successfully, THE Webhook Handler SHALL update `public.sessions.assessment` with the parsed JSON for the matching session record.
6. IF the transcript is empty, THEN THE Webhook Handler SHALL skip the OpenRouter call and leave `public.sessions.assessment` as null.
7. IF the OpenRouter call fails or returns a non-200 status, THEN THE Webhook Handler SHALL log the error and leave `public.sessions.assessment` as null without raising an HTTP error to LiveKit.

---

### Requirement 5

**User Story:** As a developer, I want `chart_client.py` to expose a `fetch_chart_data` function that the updated agent can call, so that the agent can resolve a room name to a full chart record including `detailed_report`.

#### Acceptance Criteria

1. THE chart_client module SHALL expose an async function named `fetch_chart_data` that accepts a `room_name` string parameter and returns a dict.
2. WHEN `fetch_chart_data` is called, THE chart_client SHALL make an HTTP GET request to the backend `/agent/chart-for-room` endpoint with the room name as a query parameter and the `X-Agent-Secret` header.
3. WHEN the backend returns a 200 response, THE chart_client SHALL return the full response JSON including the `detailed_report` and `full_name` fields.
4. IF the backend returns a non-200 response or the request times out, THEN THE chart_client SHALL return an empty dict.
