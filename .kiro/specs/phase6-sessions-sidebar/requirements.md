# Requirements Document

## Introduction

Phase 6 adds a Historical Sessions Sidebar to the JyotishAI frontend. After a user completes a voice consultation with Celeste AI, the session is stored in the database with an AI-generated assessment. This feature surfaces those past sessions in a scrollable sidebar panel on the session page, and lets users drill into a detailed view showing the AI assessment and full conversation transcript via a slide-in modal panel.

The sidebar integrates with the existing authenticated fetch architecture (`apiGet` from `lib/api.ts`), requires a new backend endpoint to query the `public.sessions` table by user ID, and must coexist with the existing LiveKit room layout in `app/session/page.tsx`.

## Glossary

- **Session**: A completed or active voice consultation record stored in `public.sessions`, containing `id`, `livekit_room`, `status`, `started_at`, `duration_secs`, and `assessment` (JSONB).
- **SessionAssessment**: A structured JSON object with `session_summary` (string), `key_insights` (string[]), and `action_items` (string[]), generated by OpenRouter after a session ends.
- **SessionsSidebar**: The React client component that fetches and displays the list of past sessions for the authenticated user.
- **DetailPanel**: The slide-in modal overlay rendered when a user clicks a session card, showing the AI Assessment and Full Transcript tabs.
- **Transcript**: The ordered list of `session_messages` rows (`role`, `content`) associated with a session.
- **apiGet**: The authenticated fetch utility in `lib/api.ts` that attaches the Supabase JWT to all backend requests.
- **useUser**: The React hook in `lib/hooks/useUser.ts` that returns the currently authenticated Supabase user object.
- **Backend Sessions Endpoint**: A new FastAPI route `GET /sessions/user/{user_id}` that returns all sessions for a given user, ordered by `started_at` descending.
- **LiveKitRoom**: The existing LiveKit room wrapper in `app/session/page.tsx` that must remain intact and functional.

## Requirements

### Requirement 1

**User Story:** As an authenticated user, I want to see a list of my past consultation sessions in a sidebar on the session page, so that I can review my history without leaving the app.

#### Acceptance Criteria

1. WHEN the session page loads, THE SessionsSidebar SHALL fetch the authenticated user's past sessions from the Backend Sessions Endpoint using the `apiGet` utility.
2. WHILE sessions are being fetched, THE SessionsSidebar SHALL display a loading state message in place of the session list.
3. IF the fetch returns an empty array, THEN THE SessionsSidebar SHALL display a message indicating no past sessions exist.
4. IF the fetch request fails, THEN THE SessionsSidebar SHALL display an error state message and log the error to the console.
5. WHEN sessions are loaded successfully, THE SessionsSidebar SHALL render one card per session showing the session date, duration in minutes and seconds, and a truncated session summary.

---

### Requirement 2

**User Story:** As an authenticated user, I want to click a past session card and view its full AI assessment and transcript, so that I can revisit insights from previous consultations.

#### Acceptance Criteria

1. WHEN a user clicks a session card, THE DetailPanel SHALL animate into view as a slide-in overlay from the right side of the screen.
2. THE DetailPanel SHALL display two tabs: "AI Assessment" and "Full Transcript".
3. WHILE the "AI Assessment" tab is active, THE DetailPanel SHALL display the `session_summary`, `key_insights` list, and `action_items` list from the session's `assessment` field.
4. IF the session's `assessment` field is null, THEN THE DetailPanel SHALL display a message indicating the assessment is not yet available.
5. WHILE the "Full Transcript" tab is active, THE DetailPanel SHALL render each message in the session's `messages` array as a styled chat bubble, distinguishing user messages from assistant messages visually.
6. WHEN a user clicks outside the DetailPanel or clicks the close button, THE DetailPanel SHALL animate out and be removed from the DOM.

---

### Requirement 3

**User Story:** As a developer, I want a backend endpoint that returns all sessions for a given user, so that the frontend sidebar can fetch real session data.

#### Acceptance Criteria

1. THE Backend Sessions Endpoint SHALL accept a `GET` request at the path `/sessions/user/{user_id}`.
2. THE Backend Sessions Endpoint SHALL require a valid Supabase JWT via the `get_current_user` dependency and return a 401 response if the token is absent or invalid.
3. WHEN called with a valid `user_id`, THE Backend Sessions Endpoint SHALL query `public.sessions` joined with `public.session_messages` and return sessions ordered by `started_at` descending.
4. THE Backend Sessions Endpoint SHALL return each session with fields: `id`, `livekit_room`, `status`, `started_at`, `duration_secs`, `assessment`, and a `messages` array of `{role, content}` objects.
5. IF no sessions exist for the user, THEN THE Backend Sessions Endpoint SHALL return an empty `sessions` array with a 200 status.

---

### Requirement 4

**User Story:** As an authenticated user, I want the session page layout to show the sidebar alongside the active LiveKit room, so that I can see my history while in a session.

#### Acceptance Criteria

1. THE session page SHALL render the SessionsSidebar and the LiveKit room side-by-side in a full-screen flex layout.
2. THE SessionsSidebar SHALL occupy a fixed-width column on the left side of the session page.
3. THE LiveKit room container SHALL occupy the remaining horizontal space to the right of the sidebar.
4. WHEN the authenticated user's ID is not yet available, THE SessionsSidebar SHALL not initiate a fetch request.
5. THE session page SHALL pass the authenticated user's ID from the `useUser` hook to the SessionsSidebar component.
